# 第 5 部分：什么是 MCP，为什么你应该关注？

---

## 首先，快速回顾

- **第 3 部分：** 我们了解到工具让模型能够**执行操作**。
- **第 4 部分：** 我们看到 RAG 帮助模型在回答前**找到相关信息**。

这些都是**外部支持**——它们帮助模型更智能地行动，但协调工作仍然在模型之外进行。

但如果你能以一种清晰、结构化的格式传递**模型所需的所有上下文**——工具、检索到的数据、记忆、指令——会怎么样？

这就是**模型上下文协议（MCP）**试图解决的问题。

---

## 那么什么是 MCP？
<img width="737" height="452" alt="image" src="https://github.com/user-attachments/assets/2e17fcb6-ab35-4a4c-88b4-c74db165e4b8" />


从本质上讲，**模型上下文协议**是一种标准化的方式，为 LLM 提供推理和响应所需的一切。

可以把它想象成打包：

- 你希望模型完成的任务
- 它可以使用的工具/API
- 它可能需要的文档或记忆
- 对话中的先前消息

……然后一次性交付所有这些内容。

它**不是**工具、库或产品。
它是一种**协议**——模型与外部世界之间的通信结构。

如果你来自技术领域，类似的例子有：**HTTP**、**TCP/IP** 或 **SMTP**。
如果你不是技术出身，只需记住：技术人员喜欢标准化——它使事物更容易重用和组合。

---

## 这为什么重要？

假设你正在构建一个智能体。
现在，你可能需要同时处理：

- 发送提示
- 传递检索到的文档
- 注册工具
- 管理状态
- 跟踪之前发生的事情

MCP 说：
> “让我们标准化如何将所有这些内容提供给模型，这样我们就不必为每个用例重新发明轮子。”

对于**企业**来说，这非常重要。
随着智能体变得越来越复杂，协调**工具**、**RAG**、**记忆**和**输出**变得混乱。

MCP 使这种编排**可组合**、**模块化**，并更容易插入其他系统。

如果你曾经使用过 API，可以将 MCP 视为**定义良好的请求模式**。
不是将所有内容都扔进一个长字符串并希望模型能理解，模型仍然看到文本——但它是**结构化的**，具有**清晰的上下文、选项和依据**。

---

## 为什么 MCP 如此迅速流行起来？

鉴于 MCP 只是一个协议，你可能会问：
> 它有什么优势，为什么每个人都开始使用它？

以下是原因：

1. **AI 原生** — MCP 是为 AI 智能体构建的。它为当今智能体使用的一切提供了空间：工具、提示、记忆、文档等等。
2. **完善的文档和示例** — Anthropic（MCP 的创建者）不仅发布了规范，还发布了客户端、SDK、测试工具和现实世界的演示。
3. **网络效应** — 2024 年 11 月悄然发布时，大多数人都忽略了它……直到 2025 年，它突然爆发。工具、初创公司甚至 OpenAI 都开始支持它。

---

## 常见误解

- **MCP 不是新的 API 或产品** — 它只是一种模式，一种清晰地组织发送给模型内容的方式。
- **它不会让模型更智能** — 它只是为模型提供更好、更结构化的上下文。
- **它不仅适用于智能体** — 即使是简单的助手也能从更好的上下文管理中受益。

---

## 那么……你应该关注吗？

如果你正在构建玩具提示或快速演示——可能不需要（目前）。

但如果你正在从事：

- 企业级智能体
- 多工具工作流
- 需要访问**记忆 + RAG + 规划**的 LLM
- **上下文管理**是瓶颈的系统

……那么**是的**，你应该关注。MCP 是关于更好地将不断演变的结构化上下文传递到模型中。

但请记住：MCP 只是一个协议。
像所有标准一样，只有在广泛采用的情况下它才有效。
如果在 MCP 成为“智能体的 HTTP”之前出现了更好的东西，生态系统可能会再次转变。
